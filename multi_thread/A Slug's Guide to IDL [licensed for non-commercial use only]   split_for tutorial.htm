split_for tutorial

Page history last edited by Robert da Silva 3 years ago

If you want to use the split_for.pro routine you will need the following codes for dependencies:

  • struct_pass.pro
  • rstring.pro
  • splog.pro
  • x_chkfil.pro

 

Before we begin

 

  • Code inside a child process (or this case sent to split_for) is difficult to debug. So debug all your code before translating it to split_for. Notes on tools split_for uses for debugging are below.
  • the code can be split any number of times given by the nsplit keyword (default to 2), but be aware that parallel code only increases speed if you are using more resources by running parallel. Running 100 jobs on 4 cores won't help you much and will likely run much slower. Also many IDL routines are parallel (e.g. many basic mathematical operations like '+' etc...) so be sure you want to run a parallel job before you use this program  

 

Example

Note that this example is actually a very poor use of split_for as the overheads are longer than the code to run. As a result the job still runs serial but this is a minor point. Make your loop longer and you will see the advantage.

 

Now imagine I have the following for loop: 

        kk=10

     for i=0, 10 do begin  

        if n_elements(mm) EQ 0 then mm=i else mm=[mm, i]

        if n_elements(nn) EQ 0 then nn=i^2 else nn=[nn, i^2]

     endfor

 

This can be split up with split_for with the following command:

        .f         

       kk=10

       split_for, 0, 10, commands=[$

             'if n_elements(mm) EQ 0 then mm=i else mm=[mm, i]',$

             'if n_elements(nn) EQ 0 then nn=i*kk else nn=[nn, i*kk]'], $

             varnames='kk', $

             outvar=['mm', 'nn']

After this runs, we can check out variables with a help command

     IDL> help

     % At $MAIN$          

     KK              INT       =       10

     MM0             INT       = Array[6]

     MM1             INT       = Array[5]

     NN0             INT       = Array[6]

     NN1             INT       = Array[5]

     Compiled Procedures:

         $MAIN$  SPLIT_FOR   SPLOG       XMANAGER

 

     Compiled Functions:

         RSTRING     SPLOG_TRIM  STR_SEP     X_CHKFIL

We note that we now have two new variables called mm0 amd mm1. These variables represent the outputs of the each of the splits. i.e. mm0 corresponds to the mm variable that was built up in split 0 and mm1 corresponds to the values from string 1. I leave the combination of the outputs to the user. As a note, this code will clobber any existing variables.

 

I know this sounds a little confusing so we will take some time to go through this more carefully. Let's dissect the calling sequence:

   .f         

This is just to give us a clean slate for testing purposes

   kk=10

Here we define a variable which we will pass to the for loop.

   split_for, 0, 10, commands=[$

Ok now we are calling split_for. We are setting up a loop from 0 to 10.

        'if n_elements(mm) EQ 0 then mm=i else mm=[mm, i]',$

Now we pass the first command to the for loop. This command just makes an array which stores the indices. This command can be anything you want it to be.

        'if n_elements(nn) EQ 0 then nn=i*kk else nn=[nn, i*kk]'], $

Now we pass the second command. Note how we do this. The commands are passed an array of strings where each element is a line that gets passed to the for loop.

        varnames='kk', $

This syntax passes in the variable we want to input. It is called kk. We could pass more variables by passing an array of names here. You pass the names of the variables and not the variables themselves!You can access the name of a variable with scope_varname(), for example scope_varname(kk) returns the string 'kk'. If you want to pass as structure you need to use the struct2pass syntax. 

        outvar=['mm', 'nn']

Now we name the variables that we want out of our for loop. So how do we get access to these variables? Well they will just be there after you call the program (magic isn't it). The program will create a variable at the level of the program from which you call it, so if you already have a variable of that name it will be overwritten. Since the variables can be of any type it is non-trivial for the code to figure out how you want them connected. So instead the code outputs an instance of each variable for each split it makes. In our instance, we had 2 splits so there are two version of each of the mm and nn variables. These are called mm0, mm1, nn0, nn1. mm0 represents the version of mm that was output from the first split and mm1 represent the version of mm that was output by the second. The same is the case for nn.

 

If this is unclear, let me know and I will add to this explanation.

Why doesn't it always run exactly nsplit splits?    

The code tries to run the minimum number of jobs which offer the same speed increase. The exact rule for determining how many splits there will be is below

 

    nsplit=nsplit < (finish-start+1) ;the maximum number of splits possible is the number of iterations in your loop 

    step=ceil((finish-start+1.)/nsplit)     ;this calculates the step that is achievable for maximum speed increase          

                                            ;given your start and finish for the for loop and the number of splits.

    ;----

    ; the following bit finds out the minimum number of splits that get you the most speed 

    fin_arr=start+step*(findgen(nsplit)+1)

    min_fin_arr=min(where(fin_arr GE finish))

    nsplit=min_fin_arr+1

 

 

How to quickly convert your code

Imagine you have the following for loop

for i=0, 10 do begin

     expr1

     expr2

     expr3

     expr4

endfor

 

Aside from passing variables in and out all have to do is use a macro to edit each of the lines inside the for loop to begin with an ' and end with ',$. See below.

for i=0, 10 do begin

     'expr1',$

     'expr2',$

     'expr3',$

     'expr4',$

endfor

 

Now remove the endfor line and replace the last ',$ with ']

for i=0, 10 do begin

     'expr1',$

     'expr2',$

     'expr3',$

     'expr4']

 

Now replace the first line with the proper split_for syntax:

split_for,0, 10, commands=[$

     'expr1',$

     'expr2',$

     'expr3',$

     'expr4']

 

Then add in anything you want for passing variables in and out:

split_for,0, 10, commands=[$

     'expr1',$

     'expr2',$

     'expr3',$

     'expr4'], varnames=['foo1', 'foo2'], outvar=['foo3', 'foo4']

 

And you are done! 

Note: You can enclose single quotes in double quotes for instance to include the command:

     print, 'hello world'

you can simply use

     "print, 'hello world'"

 

 

How to debug

I told you that you had to debug your code before you passed it in to split_for... but say you didn't because you are a bad person... then you can use the debug keyword. This works exactly the same as the normal split_for except you use the keyword debug. The syntax is to set the debug keyword to be the number of the split you want to debug. There remain issues with this feature and it is thus *not* recommended. This is still in development.
